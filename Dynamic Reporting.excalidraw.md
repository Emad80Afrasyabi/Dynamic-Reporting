Trino (formerly PrestoSQL) is a distributed SQL query engine. It does not store data itself—it queries data where it lives across many sources and joins them in a single query. ^fAqzYT0v

What Trino Is ^Ci2ThMYJ

Key architecture:

Coordinator: Parses SQL, plans execution, returns results.
Workers: Execute query fragments in parallel.
Connectors: Plugins to talk to PostgreSQL, Elasticsearch, Kafka, S3, BigQuery, etc.
Catalogs: Named Trino connections to a data source (e.g., postgresql, es, hive).
Schemas and tables: Trino exposes each connector’s objects as catalog.schema.table.
 ^w6gQTCZs

Performance Considerations ^WbnS9KbD

Predicate pushdown: Trino pushes WHERE, LIMIT, aggregations, and projections to connectors when possible. ^QvlKTWUx

Partitioning and indexing in source: PostgreSQL and Elasticsearch still need proper indexes. ^qT3VkAD1

Network latency: Trino pulls data from source systems; large scans are slow. ^vDkOyT4X

Memory: Trino is memory-hungry; configure JVM heap and spill-to-disk. ^ar7OaNzq

Materialized views / caching: Use Trino materialized views or a separate cache/catalog. ^SLqrMgYB

Avoid cross-joining massive datasets: Joins between PostgreSQL and Elasticsearch pull data into Trino and join in memory. ^vkCxS2or

Limitations You Should Know ^umwlaSEm

Trino does not support all SQL features across all connectors. ^1sVtaxri

Elasticsearch nested objects can be awkward in SQL. ^icAGXpBX

Write support varies by connector. ^4dl3f1v6

Transactions are not fully supported across connectors. ^jZMGzL7h

T-SQL is not natively supported—you must translate or use Trino SQL. ^hMwdp4b5
